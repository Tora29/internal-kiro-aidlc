## 通信パターン

### フロントエンド → バックエンド

#### 1. ユーザーログイン

```
認証UI
  ↓ (POST /auth/login)
API ゲートウェイ
  ↓
認証サービス
  ↓ (Cognito)
Amazon Cognito
  ↓ (JWT トークン)
認証UI
```

#### 2. 言い訳生成

```
言い訳生成UI
  ↓ (POST /excuses/generate)
API ゲートウェイ
  ↓ (認証検証)
認証サービス
  ↓
言い訳生成サービス
  ├→ (Claude Haiku/Sonnet)
  │  Amazon Bedrock
  ├→ (トークン記録)
  │  トークン管理サービス
  │    ↓ (DynamoDB)
  │    DynamoDB
  └→ (言い訳保存)
     言い訳履歴サービス
       ↓ (DynamoDB)
       DynamoDB
  ↓ (言い訳)
言い訳生成UI
```

#### 3. Pro アップグレード

```
言い訳生成UI
  ↓ (POST /plans/upgrade-to-pro)
API ゲートウェイ
  ↓ (認証検証)
認証サービス
  ↓
プラン管理サービス
  ├→ (支払い処理)
  │  AWS Marketplace
  └→ (プラン更新)
     DynamoDB
  ↓ (アップグレード完了)
言い訳生成UI
```

#### 4. 懺悔室

```
懺悔室UI
  ↓ (POST /confessions/send)
API ゲートウェイ
  ↓ (認証検証)
認証サービス
  ↓
攻撃検知サービス
  ↓ (攻撃判定)
懺悔室サービス
  ├→ (Claude Haiku)
  │  Amazon Bedrock
  └→ (肯定メッセージ)
  ↓ (肯定メッセージ)
懺悔室UI
```

#### 5. キャラクター選択

```
オンボーディング
  ↓ (POST /characters/select)
API ゲートウェイ
  ↓ (認証検証)
認証サービス
  ↓
キャラクター管理サービス
  ├→ (シーン保存)
  │  DynamoDB
  └→ (キャラクター決定)
  ↓ (キャラクター情報)
キャラクター表示
```

## 依存関係の詳細

### 強い依存関係（必須）

| 依存元 | 依存先 | 理由 |
|------|------|------|
| 言い訳生成サービス | Amazon Bedrock | 言い訳生成に必須 |
| 懺悔室サービス | Amazon Bedrock | 肯定メッセージ生成に必須 |
| 認証サービス | Amazon Cognito | ユーザー認証に必須 |
| トークン管理サービス | DynamoDB | トークン使用量保存に必須 |
| プラン管理サービス | DynamoDB | プラン情報保存に必須 |
| キャラクター管理サービス | DynamoDB | キャラクター情報保存に必須 |
| 言い訳履歴サービス | DynamoDB | 言い訳履歴保存に必須 |

### 弱い依存関係（オプション）

| 依存元 | 依存先 | 理由 |
|------|------|------|
| 言い訳生成サービス | トークン管理サービス | トークン上限チェック（フォールバック可能） |
| 言い訳生成サービス | 攻撃検知サービス | 攻撃検知（フォールバック可能） |
| プラン管理サービス | AWS Marketplace | 支払い処理（フォールバック可能） |

## 循環依存の確認

### 循環依存なし

すべてのコンポーネント間の依存関係は一方向であり、循環依存はありません。

```
認証サービス
  ↓
言い訳生成サービス
  ↓
トークン管理サービス
  ↓
プラン管理サービス
```

## 通信プロトコル

### REST API

**ベース URL**: `https://api.donmai.example.com`

**認証**: JWT トークン（Authorization ヘッダー）

**リクエスト形式**:
```json
{
  "method": "POST",
  "path": "/excuses/generate",
  "headers": {
    "Authorization": "Bearer <JWT_TOKEN>",
    "Content-Type": "application/json"
  },
  "body": {
    "situation": "今日の朝会に遅刻した"
  }
}
```

**レスポンス形式**:
```json
{
  "status": "success",
  "data": {
    "excuse": "朝の電車が遅延していました",
    "tokens_used": 150,
    "remaining_tokens": 9850
  },
  "error": null
}
```

### エラーレスポンス形式

```json
{
  "status": "error",
  "data": null,
  "error": {
    "code": "TOKEN_LIMIT_EXCEEDED",
    "message": "トークン上限に達しました"
  }
}
```

## キャッシング戦略

### ElastiCache による分散キャッシング

**キャッシュ対象**:
- ユーザー情報（TTL: 1 時間）
- プラン情報（TTL: 1 時間）
- キャラクター情報（TTL: 24 時間）

**キャッシュ無効化**:
- ユーザー情報更新時
- プラン変更時
- キャラクター選択時

**キャッシュ戦略**:
```
リクエスト
  ↓
ElastiCache チェック
  ├→ キャッシュヒット: キャッシュから返す
  └→ キャッシュミス: DynamoDB から取得してキャッシュに保存
```

## エラーハンドリング

### 集中管理エラーハンドラー

**エラーハンドリングフロー**:
```
サービス
  ↓ (エラー発生)
エラーハンドラー
  ├→ エラーログ記録
  │  └→ CloudWatch
  ├→ エラー分類
  │  ├→ 4xx: クライアントエラー
  │  └→ 5xx: サーバーエラー
  └→ エラーレスポンス生成
     ↓
     API ゲートウェイ
     ↓
     フロントエンド
```

**エラーコード**:
- `AUTHENTICATION_FAILED`: 認証失敗
- `TOKEN_LIMIT_EXCEEDED`: トークン上限超過
- `INVALID_PLAN`: 無効なプラン
- `BEDROCK_ERROR`: Bedrock エラー
- `DATABASE_ERROR`: データベースエラー
- `ATTACK_DETECTED`: 攻撃検知
