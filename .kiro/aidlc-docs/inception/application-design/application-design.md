# Application Design - どんMai

## 設計概要

「どんMai」は、レイヤーベースのアーキテクチャで設計されています。フロントエンド層、バックエンド層（マイクロサービス）、データ層に分離されています。

---

## アーキテクチャ全体図

```
┌─────────────────────────────────────────────────────────────┐
│                    フロントエンド層 (SvelteKit)              │
├─────────────────────────────────────────────────────────────┤
│ ランディング │ デモ体験 │ オンボーディング │ 言い訳生成UI   │
│ 懺悔室UI │ ポップアップ │ キャラクター │ 認証UI │ 設定UI   │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│                  API ゲートウェイ (REST)                     │
│              認証検証 / ルーティング / エラー処理             │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│              バックエンド層 (AWS Lambda - マイクロサービス)   │
├─────────────────────────────────────────────────────────────┤
│ 認証 │ 言い訳生成 │ 懺悔室 │ トークン管理 │ プラン管理      │
│ キャラクター管理 │ 攻撃検知 │ 言い訳履歴                    │
└─────────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────────┐
│                    外部サービス統合                          │
├─────────────────────────────────────────────────────────────┤
│ Amazon Cognito │ Amazon Bedrock │ AWS Marketplace           │
│ Amazon DynamoDB │ ElastiCache │ CloudWatch                  │
└─────────────────────────────────────────────────────────────┘
```

## 設計決定

### 1. コンポーネント組織方法: レイヤーベース

**理由**:
- 関心の分離が明確
- スケーラビリティが高い
- チーム間の責務が明確

**構成**:
- **フロントエンド層**: UI コンポーネント（SvelteKit）
- **バックエンド層**: ビジネスロジック（AWS Lambda）
- **データ層**: データ永続化（DynamoDB）

---

### 2. フロントエンドコンポーネント分割: 機能ベース

**理由**:
- 機能ごとの独立性が高い
- 再利用性が高い
- テストが容易

**構成**:
- **言い訳生成UI**: 言い訳生成機能
- **懺悔室UI**: 懺悔室機能
- **認証UI**: ログイン・登録機能
- **設定UI**: ユーザー設定機能
- **共通UIコンポーネント**: 再利用可能な UI コンポーネント

---

### 3. バックエンドサービス設計: マイクロサービス

**理由**:
- 各サービスが独立して開発・デプロイ可能
- スケーラビリティが高い
- 障害の隔離が可能

**構成**:
- **認証サービス**: ユーザー認証
- **言い訳生成サービス**: 言い訳生成
- **懺悔室サービス**: 肯定メッセージ生成
- **トークン管理サービス**: トークン使用量管理
- **プラン管理サービス**: プラン情報管理
- **キャラクター管理サービス**: キャラクター分岐ロジック
- **攻撃検知サービス**: 攻撃検知
- **言い訳履歴サービス**: 言い訳履歴管理

---

### 4. API 設計パターン: RESTful

**理由**:
- 標準的で理解しやすい
- HTTP メソッドが明確
- キャッシング戦略が容易

**API エンドポイント**:
- `POST /auth/login` - ログイン
- `POST /auth/signup` - 登録
- `POST /excuses/generate` - 言い訳生成
- `GET /excuses/history` - 言い訳履歴
- `POST /confessions/send` - 懺悔送信
- `GET /tokens/remaining` - 残りトークン数
- `POST /plans/upgrade-to-pro` - Pro アップグレード
- `POST /characters/select` - キャラクター選択

---

### 5. データベース設計: 正規化

**理由**:
- データの一貫性が高い
- 更新時の異常が少ない
- クエリが効率的

**テーブル構成**:
- **ユーザーテーブル**: ユーザー情報、プラン情報、シーン選択
- **トークン使用量テーブル**: ユーザー ID、日付、使用トークン数
- **言い訳履歴テーブル**: ユーザー ID、言い訳テキスト、生成日時

---

### 6. 認証・認可設計: Cognito のみ

**理由**:
- AWS ネイティブサービス
- セキュリティが高い
- 管理が簡単

**構成**:
- **Cognito ユーザープール**: ユーザー管理
- **JWT トークン**: API 認証
- **Google ソーシャルログイン**: オンボーディング離脱を減らす

---

### 7. キャッシング戦略: 分散キャッシング

**理由**:
- パフォーマンスが向上
- スケーラビリティが高い
- 複数インスタンス間でキャッシュを共有

**構成**:
- **ElastiCache**: ユーザー情報、プラン情報、キャラクター情報をキャッシュ
- **TTL**: 1 時間（ユーザー情報、プラン情報）、24 時間（キャラクター情報）

---

### 8. エラーハンドリング設計: 集中管理

**理由**:
- エラー処理が一貫性を持つ
- ログ記録が統一される
- デバッグが容易

**構成**:
- **中央エラーハンドラー**: すべてのエラーを集中管理
- **エラーログ**: CloudWatch に記録
- **エラーレスポンス**: 標準形式で返す

---

### 9. ロギング・監視設計: CloudWatch のみ

**理由**:
- AWS ネイティブサービス
- 統合が簡単
- コストが低い

**構成**:
- **CloudWatch Logs**: アプリケーションログ
- **CloudWatch Metrics**: カスタムメトリクス
- **CloudWatch Alarms**: アラート設定

---

### 10. 非同期処理設計: ハイブリッド

**理由**:
- 同期処理で即座のレスポンスが必要な場合に対応
- 非同期処理でバックグラウンド処理が可能

**構成**:
- **同期処理**: 言い訳生成、ログイン、プラン変更
- **非同期処理**: ログ記録、メトリクス記録、支払い処理

## 主要なデータフロー

### 1. ユーザーオンボーディング

```
ユーザー
  ↓
ランディングページ
  ↓ (デモボタン)
デモ体験
  ├→ 言い訳生成UI
  │  ├→ 言い訳生成サービス
  │  │  └→ Amazon Bedrock
  │  └→ キャラクター表示
  └→ アカウント作成
     ↓
     認証UI
     ├→ 認証サービス
     │  └→ Amazon Cognito
     └→ プラン選択
        ↓
        オンボーディング
        ├→ シーン選択
        │  ↓
        │  キャラクター管理サービス
        │  └→ DynamoDB
        └→ キャラクター表示
           ↓
           ホーム画面
```

### 2. 言い訳生成

```
ユーザー
  ↓
言い訳生成UI
  ├→ 状況入力
  └→ 生成ボタン
     ↓
     言い訳生成サービス
     ├→ 攻撃検知
     │  └→ 攻撃検知サービス
     ├→ 言い訳生成
     │  └→ Amazon Bedrock
     ├→ トークン記録
     │  └→ トークン管理サービス
     │     └→ DynamoDB
     └→ 言い訳保存
        └→ 言い訳履歴サービス
           └→ DynamoDB
     ↓
     言い訳表示
```

### 3. Pro アップグレード

```
ユーザー
  ↓
言い訳生成UI
  ├→ トークン上限超過
  └→ Pro 誘導モーダル
     ↓
     Pro 登録ボタン
     ↓
     プラン管理サービス
     ├→ 支払い処理
     │  └→ AWS Marketplace
     └→ プラン更新
        └→ DynamoDB
     ↓
     Pro ユーザー
```

## 非機能要件への対応

### パフォーマンス（3 秒以内の応答）

- **キャッシング**: ElastiCache による分散キャッシング
- **非同期処理**: ログ記録、メトリクス記録を非同期化
- **Lambda 最適化**: メモリ設定、タイムアウト設定

### セキュリティ

- **認証**: Amazon Cognito + JWT トークン
- **認可**: API ゲートウェイで JWT 検証
- **攻撃検知**: プロンプトインジェクション対策、キーワード検知

### スケーラビリティ

- **Lambda オートスケーリング**: 同時実行数 1000
- **DynamoDB オンデマンド**: 自動スケーリング
- **ElastiCache**: 分散キャッシング

## 依存関係の管理

### 強い依存関係（必須）

- 言い訳生成サービス ← Amazon Bedrock
- 認証サービス ← Amazon Cognito
- すべてのサービス ← DynamoDB

### 弱い依存関係（オプション）

- 言い訳生成サービス ← トークン管理サービス（フォールバック可能）
- 言い訳生成サービス ← 攻撃検知サービス（フォールバック可能）
